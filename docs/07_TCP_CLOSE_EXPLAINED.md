# TCP 連接關閉詳解

## 🔌 TCP 連接的兩個通道

TCP 連接實際上有**兩個獨立的通道**：

```
客戶端                                伺服器
   │                                    │
   │ ────────→ 寫通道（發送）───────→   │  讀通道（接收）
   │                                    │
   │ ←──────── 讀通道（接收）←─────────  │  寫通道（發送）
   │                                    │
```

### 兩種關閉方式

#### 1. shutdown() - 部分關閉（半關閉）

```c
// 只關閉寫通道
shutdown(fd, SHUT_WR);
// 結果：
//   ✅ 讀通道仍然開啟（可以接收）
//   ❌ 寫通道關閉（不能發送）

// 只關閉讀通道
shutdown(fd, SHUT_RD);
// 結果：
//   ❌ 讀通道關閉（不能接收）
//   ✅ 寫通道仍然開啟（可以發送）

// 關閉讀寫
shutdown(fd, SHUT_RDWR);
// 結果：
//   ❌ 讀寫都關閉
//   ⚠️ 但 fd 仍然存在，需要 close()
```

#### 2. close() - 完全關閉

```c
close(fd);
// 結果：
//   ❌ 讀通道關閉
//   ❌ 寫通道關閉
//   ❌ fd 被釋放
```

## 🎬 您的問題：當 EOF 時發生了什麼

### 完整的關閉時序

```
步驟    客戶端 (nc)                      伺服器
──────────────────────────────────────────────────────
1      用戶按空行 Ctrl+D
       ↓
       shutdown(SHUT_WR)                  
       只關閉寫通道 ─────────────────────→
       
       狀態：
       ✅ 可以接收（讀通道開啟）
       ❌ 不能發送（寫通道關閉）
                                          recv() 返回 0
                                          檢測到 EOF
                                          ↓
2                                         closeClient(idx)
                                          ↓
                                          close(fd)
                                          關閉讀寫兩端 ←───────────
       
       收到 FIN ←────────────────────────
       
       狀態：
       ❌ 不能接收（對端關閉）
       ❌ 不能發送（已經關閉）
       
3      Connection closed
       nc 程序結束
```

### 為什麼我們使用 close() 而不是 shutdown()？

讓我看看我們的代碼：

```cpp
// Client::closeNow()
void Client::closeNow() {
    if (_fd >= 0) {
        ::close(_fd);  // ← 使用 close()，完全關閉
        _fd = -1;
    }
}
```

**我們使用 `close(fd)`，這會同時關閉讀和寫！**

## 🤔 為什麼不只關閉讀端？

### 理論上可以實現半關閉

```cpp
// 理論上的半關閉實現
if (recv() == 0) {  // 客戶端關閉了寫端
    // 只關閉伺服器的讀端
    shutdown(fd, SHUT_RD);
    
    // 保持寫端開啟
    // 繼續發送待發的數據
    
    // 等所有數據發完後才 close(fd)
}
```

### 但我們選擇完全關閉，原因是：

#### 1. IRC 協議特性

```
IRC 是雙向互動協議：

客戶端發送命令 → 伺服器處理 → 伺服器回應
       ↓                              ↓
    需要寫端                        需要讀端

如果客戶端不能發送命令（寫端關閉）
→ 保持連接沒有實際意義
```

#### 2. 實際行為觀察

**您之前觀察到的：** "A EOF 之後還能收到訊息"

這不是因為我們實現了半關閉，而是因為**延遲關閉**：

```cpp
// Server::run() 的處理順序
for (;;) {
    poll(...);
    
    // 處理所有事件
    for (每個 fd) {
        if (有事件) {
            processPollEvent(i, toAddFds, toCloseIdx);
            // ← A 收到 EOF，只是標記到 toCloseIdx
            // ← B 發送消息給 A，A->sendLine(...)
            // ← A 的 POLLOUT 可能被觸發，send() 給 A
        }
    }
    
    // 本輪處理完畢後才真正關閉
    for (每個待關閉的 idx) {
        closeClient(idx);  // ← 這時才 close(fd)
    }
}
```

**時間窗口：**
```
T0: A 的 POLLIN 事件 → recv EOF → 標記關閉
T1: B 的 POLLIN 事件 → 發送給 A → sendLine(A)
T2: A 的 POLLOUT 事件 → send() 給 A ✅（A 還能收到！）
T3: 循環結束 → closeClient(A) → close(fd) ❌（現在才真正關閉）
```

**所以 A 短暫地能收到消息，是因為：**
- 延遲關閉（本輪處理完才關）
- 不是因為我們實現了半關閉
- `close(fd)` 仍然是完全關閉讀寫

## 📊 close() vs shutdown() 對比

### 我們當前的實現

```cpp
void Client::closeNow() {
    ::close(_fd);  // 完全關閉
}
```

**效果：**
```
伺服器端：
  ❌ 讀通道關閉（不能再 recv）
  ❌ 寫通道關閉（不能再 send）
  ❌ fd 釋放

客戶端收到 FIN：
  ❌ 連接完全終止
```

### 如果實現半關閉（理論）

```cpp
void Client::closeNow() {
    // 選項 1: 只關閉讀端
    ::shutdown(_fd, SHUT_RD);  // 半關閉
    
    // 選項 2: 只關閉寫端
    ::shutdown(_fd, SHUT_WR);  // 半關閉
}
```

**但問題是：**
- 需要額外的狀態追蹤（哪些 fd 是半關閉的）
- 需要在適當時機調用 `close()` 完全關閉
- 增加複雜度
- 對 IRC 沒有實際價值

## 🎯 正確的理解

### 當空行 Ctrl+D 時

```
時刻 0: 客戶端按 Ctrl+D
───────────────────────────────────────────────
客戶端狀態：
  nc 執行 shutdown(SHUT_WR)
  ✅ 客戶端可以接收（讀通道開啟）
  ❌ 客戶端不能發送（寫通道關閉）

伺服器狀態：（此時還沒處理）
  ✅ 可以接收（讀通道開啟）
  ✅ 可以發送（寫通道開啟）


時刻 1: 伺服器 recv() 返回 0
───────────────────────────────────────────────
伺服器檢測到 EOF
標記為待關閉


時刻 2: 本輪 poll 處理中（可能）
───────────────────────────────────────────────
其他客戶端發送消息給這個客戶端
伺服器 send() 給客戶端 ✅
客戶端收到 ✅（因為客戶端的讀通道仍開啟）


時刻 3: closeClient() 執行
───────────────────────────────────────────────
伺服器：
  close(fd)  ← 完全關閉讀寫
  
客戶端：
  收到 FIN
  讀通道也關閉
  Connection closed
```

### 為什麼用 close() 而不是 shutdown(SHUT_RD)？

#### 理由 1：簡化狀態管理

```cpp
// 使用 close() - 簡單
if (recv() == 0) {
    closeClient(idx);  // 完全關閉，清理資源
    // fd 釋放
    // 從 _pfds 移除
    // 從 _clients 移除
    // 完成 ✓
}

// 使用 shutdown(SHUT_RD) - 複雜
if (recv() == 0) {
    shutdown(fd, SHUT_RD);  // 只關閉讀
    
    // 需要追蹤狀態：
    // - 這個 fd 是半關閉的
    // - 不要再監聽 POLLIN
    // - 繼續監聽 POLLOUT
    // - 等 _outbuf 清空後才 close()
    // - 如果一直發不出去怎麼辦？
    // - 設定超時？
    
    // 複雜！
}
```

#### 理由 2：IRC 協議特性

```
IRC 是請求-回應協議：
客戶端：PRIVMSG #test :hi
伺服器：處理並廣播

如果客戶端不能發送命令（EOF）
→ 保持連接的意義？
→ 伺服器不會主動推送（除了頻道消息）
→ 客戶端無法互動
→ 沒有實際用途
```

#### 理由 3：資源效率

```
半關閉需要：
- 佔用 fd（有限資源）
- 保留在 poll 陣列中
- 額外的狀態追蹤
- 可能的超時機制

完全關閉：
- 立即釋放 fd
- 從 poll 陣列移除
- 簡單清理
- 資源立即可用
```

## 💡 為什麼 A 還能短暫收到消息？

**不是因為半關閉，而是因為延遲關閉！**

### 關鍵代碼

```cpp
// Server::run()
void Server::run() {
    for (;;) {
        poll(...);
        
        // ========== 處理事件階段 ==========
        for (size_t i = 0; i < originalCount; ++i) {
            if (_pfds[i].revents) {
                processPollEvent(i, toAddFds, toCloseIdx);
            }
        }
        // 此時：
        // - A 已經標記到 toCloseIdx
        // - 但 A 的 fd 還沒 close()
        // - A 的 Client* 還在 _clients 中
        // - A 仍然在 _pfds 中
        
        // ========== 關閉階段 ==========
        for (size_t k = 0; k < toCloseIdx.size(); ++k) {
            closeClient(idx);  // ← 這時才 close(fd)
        }
    }
}
```

### 時間窗口

```
T0: poll() 返回，發現多個事件
────────────────────────────────────────
事件 1: A 的 POLLIN
  recv() = 0  ← A 發送了 EOF
  toCloseIdx.push_back(A)  ← 只是標記！
  
事件 2: B 的 POLLIN  
  "PRIVMSG alice :bye"
  找到 A
  A->sendLine(...)  ← A 的 _outbuf 有數據
  enableWriteForFd(A)
  
事件 3: A 的 POLLOUT（可能在本輪或下輪）
  send(A_fd, ...)  ← 發送給 A
  A 收到！✅（因為 A 的讀通道仍開啟，且伺服器還沒 close）

所有事件處理完畢（通常 < 1 毫秒）
────────────────────────────────────────
T1: 執行關閉
  closeClient(A)
  ↓
  close(A_fd)  ← 現在才完全關閉！
  ↓
  伺服器發送 FIN 給 A
  ↓
  A 收到 FIN
  ↓
  Connection closed
```

**所以：**
- A 能短暫收到消息是因為**延遲關閉**
- 不是因為我們實現了半關閉
- `close()` 仍然是完全關閉讀和寫

## 📋 close() 的實際效果

### 在我們的代碼中

```cpp
// src/Client.cpp
void Client::closeNow() {
    if (_fd >= 0) {
        ::close(_fd);  // ← 這個調用做了什麼？
        _fd = -1;
    }
}
```

### close() 做了什麼

```c
close(fd) 的效果：

1. 關閉文件描述符
   - fd 號碼被釋放
   - 可以被其他 socket 重用

2. 如果這是 socket 的最後一個引用：
   - 關閉讀通道（不能再 recv）
   - 關閉寫通道（不能再 send）
   - 發送 TCP FIN 給對端
   
3. 等待對端確認（TCP 四次揮手）
```

### TCP 四次揮手

```
伺服器              客戶端
   │                   │
   │  ── FIN ────→     │  伺服器：「我要關閉了」
   │                   │
   │  ←── ACK ────     │  客戶端：「我知道了」
   │                   │
   │  ←── FIN ────     │  客戶端：「我也要關閉」
   │                   │
   │  ── ACK ────→     │  伺服器：「好的」
   │                   │
  關閉完成          關閉完成
```

**當我們調用 `close(fd)` 時，這個過程自動發生。**

## 🎯 回答您的問題

### Q: "EOF 時為何只關閉客戶端，不是讀寫兩端都關？"

**A:** 實際上我們**確實是讀寫兩端都關閉**了！

```cpp
close(fd);  // ← 這會關閉讀和寫兩端
```

**但在關閉之前：**
1. A 發送 EOF（客戶端的寫端關閉）
2. 伺服器標記 A 為待關閉（還沒真正關）
3. 處理完本輪其他事件（可能包括發送給 A）
4. **然後** `close(fd)`（伺服器的讀寫都關）

### 為什麼 A 還能短暫收到消息？

**不是因為我們只關了一端，而是因為：**

```
A 的讀通道狀態時序：

T0: 客戶端執行 shutdown(SHUT_WR)
    客戶端的寫 ❌
    客戶端的讀 ✅  ← A 的讀仍然開啟

T1: 伺服器標記 A 關閉（但還沒 close）
    伺服器的讀 ✅  ← 伺服器還能檢查事件
    伺服器的寫 ✅  ← 伺服器還能發送

T2: 伺服器 send() 給 A
    數據通過 socket
    A 收到 ✅  ← A 的讀通道仍開啟

T3: 伺服器 close(fd)
    伺服器的讀 ❌  ← 現在關閉
    伺服器的寫 ❌  ← 現在關閉
    發送 FIN 給 A
    
T4: A 收到 FIN
    A 的讀 ❌  ← 現在關閉
    Connection closed
```

## 🔑 總結

| 問題 | 答案 |
|------|------|
| **我們用 close() 還是 shutdown()？** | `close(fd)` - 完全關閉 |
| **close() 關閉幾端？** | 兩端都關（讀 + 寫） |
