# 基礎概念：從零開始理解 IRC 伺服器

> 本文件專門為不熟悉網路編程的人設計，用簡單的語言解釋核心概念。

---

## 🏠 比喻：伺服器就像郵局

想像我們的 IRC 伺服器是一個**郵局**：

```
郵局（伺服器）
    ├── 櫃台人員（Listener）：迎接新客戶
    ├── 多個郵箱（Client）：每個客戶一個
    ├── 郵局管理員（Server）：協調所有工作
    └── 信件分發員（poll）：檢查哪個郵箱有新信
```

---

## 📞 什麼是 Socket？

### 簡單比喻
**Socket = 電話線**

想像每個客戶端與伺服器之間都有一條「電話線」，這就是 socket。

### 完整架構圖

```
                    伺服器 (Server)
        ┌───────────────────────────────────┐
        │                                   │
        │  ┌─────────────────────────────┐  │
        │  │   Listener (監聽 Socket)    │  │
        │  │   監聽端口 6667              │  │
        │  │   等待新連接...              │  │
        │  └─────────────────────────────┘  │
        │            ↓   ↓   ↓               │
        │    ┌───────┘   │   └───────┐       │
        │    ↓           ↓           ↓       │
        │ [Socket A] [Socket B] [Socket C]   │
        │ (Client A) (Client B) (Client C)   │
        └────│──────────│──────────│─────────┘
             │          │          │
             ↓          ↓          ↓
        客戶端 A    客戶端 B    客戶端 C
```

**重點理解：**
- **Listener** 是伺服器內部的一個特殊 socket
- Listener **不是**客戶端，而是伺服器的「門口」
- 客戶端連接到 Listener 的端口（如 6667）
- Listener 接受連接後，**創建新的 socket** 與客戶端通訊

### 連接過程

```
步驟 1: 伺服器啟動
┌─────────────┐
│   Server    │
│  ┌────────┐ │
│  │Listener│ │ ← 監聽 0.0.0.0:6667
│  │  6667  │ │    (等待連接...)
│  └────────┘ │
└─────────────┘

步驟 2: 客戶端 A 連接
客戶端 A ─────connect to 6667────→ Listener
                                    ↓
                                 accept()
                                    ↓
                          創建新 socket (fd=4)
┌─────────────┐
│   Server    │
│  ┌────────┐ │
│  │Listener│ │ ← 仍在監聽
│  └────────┘ │
│  [Socket 4]←┼──→ 客戶端 A (專用連接)
└─────────────┘

步驟 3: 客戶端 B 也連接
客戶端 B ─────connect to 6667────→ Listener
                                    ↓
                                 accept()
                                    ↓
                          創建新 socket (fd=5)
┌─────────────┐
│   Server    │
│  ┌────────┐ │
│  │Listener│ │ ← 繼續監聽更多連接
│  └────────┘ │
│  [Socket 4]←┼──→ 客戶端 A
│  [Socket 5]←┼──→ 客戶端 B
└─────────────┘
```

### 在我們的程式碼中

```cpp
// 1. 創建監聽 socket (Listener)
int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
bind(listen_fd, ...);   // 綁定到端口 6667
listen(listen_fd, ...); // 開始監聽

// 2. 接受新連接（由 Listener 處理）
int client_fd = accept(listen_fd, ...);  
// client_fd 是新創建的 socket，用來與這個客戶端通訊

// 3. 之後所有通訊都通過 client_fd
recv(client_fd, ...);  // 從客戶端讀取
send(client_fd, ...);  // 發送給客戶端
```

### Socket 的兩種類型

1. **監聽 Socket（櫃台總機）**
   - **只有一個**（整個伺服器共用）
   - **伺服器內部的組件**
   - 專門接聽新客戶的「來電」
   - 不傳輸實際數據
   - 對應我們的 `Listener`
   - 代碼中：`_listener.getFd()`

2. **連接 Socket（個別分機）**
   - **每個客戶端一個**
   - 由 Listener 的 `accept()` 創建
   - 用來與特定客戶通訊
   - 傳輸實際的命令和回應
   - 對應我們的 `Client`
   - 代碼中：每個 `Client` 對象的 `_fd`

## 🔧 什麼是 fcntl？

### 簡單比喻
**fcntl = 設定電話線的工作模式**

想像電話有兩種接聽模式：

#### 模式 1：阻塞模式（預設）
```
📞 鈴~~~ 鈴~~~
😴 我坐在這裡等電話響...
   等...
   等...
   等...
   終於有電話！接起來！
```
**問題：** 等電話時什麼事都不能做！

#### 模式 2：非阻塞模式（我們要的）
```
📞 有電話嗎？
   → 沒有：好，我去做別的
   → 有：接起來處理
     
📞 有電話嗎？（持續檢查）
   → 沒有：好，我去做別的
   → 有：接起來處理
```
**優點：** 可以同時處理多件事！

### 在我們的程式碼中

```cpp
// 設定 socket 為非阻塞模式
fcntl(fd, F_SETFL, O_NONBLOCK);

// 就像說：「這條電話線，不要讓我傻等！」
```

### 為什麼重要？

如果沒有非阻塞模式：
```
客戶端 A 正在發送很長的訊息...
伺服器在等 A 說完 😴
→ 客戶端 B、C、D 都被忽略了！❌
```

有了非阻塞模式：
```
客戶端 A 發了一部分
伺服器：收到，先存著
伺服器：檢查 B 有沒有消息
伺服器：檢查 C 有沒有消息
伺服器：回來繼續處理 A
✅ 公平對待所有客戶端！
```

## 📊 什麼是 poll？

### 簡單比喻
**poll = 快遞員的巡邏路線**

想像快遞員（poll）負責巡邏所有郵箱：

```
快遞員巡邏中...

郵箱 1（fd=4）：[有新信！] ← 通知！
郵箱 2（fd=5）：[空的]
郵箱 3（fd=6）：[有新信！] ← 通知！
郵箱 4（fd=7）：[空的]

快遞員報告：「郵箱 1 和 3 有新信！」
```

### 為什麼需要 poll？

**沒有 poll 的話：**
```cpp
// 笨方法：輪流檢查每個客戶端
for (每個客戶端) {
    recv(fd, ...);  // 有訊息嗎？
                    // 沒有就一直等... 😴
}
// ❌ 問題：如果第一個客戶端沒訊息，就卡住了！
```

**有 poll 的話：**
```cpp
// 聰明方法：一次檢查所有
poll(...);  // 告訴我哪些客戶端有訊息
// poll 返回：「fd=4 和 fd=6 有訊息！」

// 只處理有訊息的
處理 fd=4
處理 fd=6
// ✅ 不浪費時間！
```

### poll 的工作原理

```cpp
struct pollfd {
    int fd;        // 文件描述符（電話號碼）
    short events;  // 我想知道什麼事件？
                   // POLLIN  = 有人打電話進來
                   // POLLOUT = 可以打電話出去
    short revents; // 實際發生的事件（由 poll 填寫）
};

// 建立監聽清單
pollfd fds[10];
fds[0].fd = 4;  fds[0].events = POLLIN;
fds[1].fd = 5;  fds[1].events = POLLIN;
// ...

// 開始監聽！
int ready = poll(fds, 10, -1);  // -1 = 無限等待
// poll 會「睡著」，直到有事件發生

// poll 醒來了！有 ready 個事件
for (每個 fd) {
    if (fds[i].revents & POLLIN) {
        // 這個 fd 有新資料！
    }
}
```

## 🔄 我們的 run() 函數如何運作

### 完整流程圖

```
                    啟動伺服器
                        ↓
            ┌───────────────────────┐
            │   for (;;) 無限循環    │
            └───────────────────────┘
                        ↓
            ┌───────────────────────┐
            │  poll() 等待事件      │ ← 睡覺，直到有事發生
            │  所有客戶端 fd 都在這  │
            └───────────────────────┘
                        ↓
                  有事件發生！
                        ↓
            ┌───────────────────────┐
            │  檢查每個 fd          │
            └───────────────────────┘
                        ↓
        ┌───────────────┼───────────────┐
        │               │               │
    [新連接]        [可讀取]        [可寫入]
        ↓               ↓               ↓
    accept()        recv()          send()
    建立新客戶        讀取命令         發送回應
        │               │               │
        └───────────────┴───────────────┘
                        ↓
                  處理完畢
                        ↓
                  回到 poll()
```

### 詳細步驟解說

#### 步驟 1：初始化

```cpp
Server::Server(int port, const string& password) {
    // 1. 創建監聽 socket（總機）
    _listener = Listener(port);
    
    // 2. 把總機加入 poll 監聽清單
    pollfd p;
    p.fd = _listener.getFd();      // 總機的號碼
    p.events = POLLIN;              // 監聽「有新客戶」
    _pfds.push_back(p);             // 加入清單
    
    // 3. 設為非阻塞
    fcntl(p.fd, F_SETFL, O_NONBLOCK);
}
```

**比喻：**
- 郵局開門了
- 櫃台準備好了
- 開始營業！

#### 步驟 2：主循環開始

```cpp
void Server::run() {
    cout << "伺服器開始運行..." << endl;
    
    for (;;) {  // 無限循環（永遠營業）
        // 下一步...
```

#### 步驟 3：等待事件（poll）

```cpp
        // 所有 fd 都在 _pfds 陣列中
        // 可能有：
        // _pfds[0] = listener（總機）
        // _pfds[1] = 客戶端 A
        // _pfds[2] = 客戶端 B
        // _pfds[3] = 客戶端 C
        
        int ready = poll(&_pfds[0], _pfds.size(), -1);
        //              ↑           ↑              ↑
        //              陣列        多少個 fd      無限等待
        
        // poll() 在這裡「睡覺」
        // 當有任何 fd 發生事件時，poll() 醒來
        // ready = 有多少個 fd 有事件
```

**比喻：**
```
快遞員坐下來休息 😴
突然！
📬 郵箱 1 有新信！
📬 郵箱 3 也有新信！
快遞員醒來：「有 2 個郵箱需要處理！」
```

#### 步驟 4：處理事件

```cpp
        // 遍歷所有 fd
        for (size_t i = 0; i < _pfds.size(); ++i) {
            
            if (_pfds[i].revents == 0) {
                continue;  // 這個 fd 沒事件，跳過
            }
            
            // 這個 fd 有事件！
            int fd = _pfds[i].fd;
            
            // 判斷是什麼事件...
```

##### 事件類型 A：新客戶來了（POLLIN on listener）

```cpp
            if ((_pfds[i].revents & POLLIN) && 
                fd == _listener.getFd()) {
                
                // 總機響了！有新客戶！
                int new_fd = accept(...);
                
                // 把新客戶加入監聽清單
                pollfd np;
                np.fd = new_fd;
                np.events = POLLIN;  // 監聽客戶的訊息
                _pfds.push_back(np);
                
                // 設為非阻塞
                fcntl(new_fd, F_SETFL, O_NONBLOCK);
            }
```

**比喻：**
```
🔔 門鈴響了！
新客戶進門
櫃台：「您好！這是您的郵箱號碼」
把客戶的郵箱加入巡邏清單
```

##### 事件類型 B：客戶有話說（POLLIN on client）

```cpp
            else if (_pfds[i].revents & POLLIN) {
                // 某個客戶有訊息要讀
                
                char buf[4096];
                int n = recv(fd, buf, sizeof(buf), 0);
                
                if (n > 0) {
                    // 讀到資料了！
                    處理這個訊息();
                }
                else if (n == 0) {
                    // 客戶掛斷電話（EOF）
                    關閉這個連接();
                }
                else if (errno == EAGAIN) {
                    // 暫時沒資料，沒關係
                    // 下次 poll 再檢查
                }
            }
```

**比喻：**
```
📬 郵箱 2 有新信！
打開郵箱
讀取信件內容
「PRIVMSG #test :Hello」
處理這個命令
```

##### 事件類型 C：可以回信了（POLLOUT）

```cpp
            else if (_pfds[i].revents & POLLOUT) {
                // 可以發送資料了
                
                int n = send(fd, outbuf, size, 0);
                
                if (n > 0) {
                    // 發送成功
                    移除已發送的部分();
                }
                else if (errno == EAGAIN) {
                    // 暫時發不出去
                    // 下次 poll 再試
                }
                
                if (發送完畢) {
                    // 關閉 POLLOUT 監聽
                    _pfds[i].events &= ~POLLOUT;
                }
            }
```

**比喻：**
```
✉️ 可以寄信給客戶 B 了！
把回信放入郵箱
發送成功 ✅
如果郵箱滿了（EAGAIN），稍後再試
```

#### 步驟 5：清理與重複

```cpp
        }  // for 循環結束
        
        // 處理完所有事件
        // 回到 poll() 繼續等待
        
    }  // 無限循環
}
```

## 🤔 常見問題

### Q1: 為什麼要用 poll 而不是多執行緒？

**A:** 
```
多執行緒方案：
- 每個客戶端一個執行緒
- 100 個客戶端 = 100 個執行緒
- 記憶體消耗大
- 執行緒切換慢
- 複雜的鎖機制

poll 方案：
- 單一執行緒
- 100 個客戶端 = 100 個 fd 在一個陣列
- 記憶體消耗小
- 無執行緒切換
- 無需鎖
```

### Q2: 什麼是「非阻塞」？

**A:** 用餐廳點餐比喻：

```
阻塞模式（傳統餐廳）：
你：「我要一份牛排」
廚房開始做...
你站在櫃台等 😴
30 分鐘後...
你：終於做好了！
→ 期間不能服務其他客人

非阻塞模式（快餐店）：
你：「我要一份牛排」
廚房：「好的，30 分鐘後來拿」
你：去服務下一個客人
你：去服務下一個客人
...
30 分鐘後
你：檢查，牛排好了！
你：拿給客人
→ 可以同時服務多個客人
```

### Q3: EAGAIN 是什麼？

**A:** EAGAIN = "現在不行，稍後再試"

```
情境 1：讀取時
recv() 返回 EAGAIN
→ 意思：「暫時沒有資料，但連接沒斷」
→ 不是錯誤！
→ 下次 poll() 再檢查

情境 2：寫入時  
send() 返回 EAGAIN
→ 意思：「對方的接收緩衝區滿了，稍後再發」
→ 不是錯誤！
→ 下次 poll() 再試
```

### Q4: 為什麼只能有一個 poll()？

**A:**
```
多個 poll() 的問題：
poll1() 監聽 fd 1-50 😴
poll2() 監聽 fd 51-100 😴

→ 如果 poll1 的 fd 都沒事件
   poll1 會一直等待
   poll2 的 fd 就被忽略了！

一個 poll() 的好處：
poll() 監聽所有 fd 😴
→ 任何一個有事件就醒來
→ 公平處理所有客戶端
```

## 📝 總結：run() 的本質

```
run() 就是一個「事件處理員工」：

1. 坐在辦公室（進入循環）
2. 等電話響（poll 睡眠）
3. 電話響了！（poll 醒來）
4. 看是誰打來的（檢查 revents）
5. 處理這通電話（recv/send）
6. 繼續等下一通電話（回到 poll）

關鍵特點：
- 永遠不休息（無限循環）
- 公平對待所有來電（單一 poll）
- 不會傻等（非阻塞）
- 高效處理（事件驅動）
```

## 🎯 給組員的一句話總結

> **我們的伺服器用一個 poll() 監聽所有客戶端，就像一個快遞員巡邏所有郵箱。當有郵箱（客戶端）有新信（訊息）時，poll 就通知我們去處理。所有 socket 都設為非阻塞，所以我們不會卡在等待某個客戶端，可以公平服務所有人。**
