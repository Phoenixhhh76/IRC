## Why test “segmented receive”

TCP does not guarantee that “one application-layer send = one recv on the other side”. Data may arrive split into multiple chunks (low bandwidth, Nagle’s algorithm, MTU limits, kernel buffers, etc. can all cause segmentation).

IRC commands are terminated by a newline (CRLF or LF). The server must buffer fragments received across multiple recv calls, and only parse once it encounters the line ending.

The example com^Dman^Dd is simulating that "command\n" arrives in three pieces: 'com', 'man', and 'd\n'. Your server needs to reassemble the full command across multiple recvs before processing it.

## Why Ctrl+D can produce “segmentation”

The terminal is in “canonical mode” (default). Ctrl+D is the end-of-file (EOF) control character for line input:

- If the current line already has input (for example you typed com), pressing Ctrl+D will send this line immediately to nc, without appending a newline, and it will not close the connection. This creates a small chunk that sends only com.

- If the current line is empty and you press Ctrl+D, that is the real EOF (read returns 0). nc will perform a write half-close on the socket (after that, it can’t send any more).

So with the keystroke sequence:
com Ctrl+D man Ctrl+D d Enter, nc will send, in order:
- "com"
- "man"
- "d
" (-C converts Enter to CRLF)

This splits one command into three sends, simulating network fragmentation.

## Does your current code already satisfy this requirement?

Client::readFromSocket() loops on recv and appends the data to _inbuf.

popLine() only returns a complete line when it has read a line ending (CRLF or LF).

This is exactly the “aggregate fragments, parse by line” approach, so it should pass this test.

## How to test in practice

Connect:

nc -C 127.0.0.1 6667

Input (each ^D means press Ctrl+D once):
- PASS abc ⏎
- NICK nick ⏎
- USER user 0 * :hi ⏎
- JOIN #test ⏎
- com^Dman^Dd ⏎

Expected: the server should not parse the command after the first two fragments. Only when the final d⏎ (or d + Enter) arrives should it treat the full line "command" as a single instruction.

## Tips

- Pressing Ctrl+D in the middle of a line sends the current buffer but does not close. Pressing Ctrl+D on an empty line is the real EOF and will prevent you from sending anything afterward (nc does a write half-close). So when testing segmentation, make sure you press Ctrl+D while there is text on the line.
- If you want to see the segmentation more clearly, you can temporarily print the number of bytes returned by each recv inside readFromSocket().

In short: the reason the assignment asks you to use Ctrl+D is that it uses the terminal’s “end-of-line” behavior to manually create the scenario where “one command arrives in multiple deliveries”, to verify that your server correctly buffers + reassembles + parses by line.
